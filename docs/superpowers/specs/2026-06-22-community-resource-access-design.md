# Community Gate 设计说明

## 1. 项目目标

在 2026-06-26 10:00 前交付一个中国大陆网络可访问的 Web 应用，满足以下核心行为：

- 用户可以注册、登录和退出登录。
- 新注册用户默认可以访问资源 A，默认不能访问资源 B。
- 提供一个可以访问资源 B 的特权测试账号。
- 注册时调用真实 LLM 判断注册名是否违反社区规则。
- 提供公网 HTTPS URL 和一份覆盖题目要求的 README。
- README 必须记录真实的 LLM 判断效果调试、优化和复测过程。

资源 A 定义为“社区公开资料库”，资源 B 定义为“社区内部运营数据”。两者均为本项目内的 Mock 内容，但访问结果必须由后端授权策略决定。

## 2. 范围

### 2.1 本次实现

- 用户名和密码注册、登录、退出登录。
- LLM 注册名审核。
- 数据库支持的可撤销会话。
- MEMBER 与 PRIVILEGED 两种角色。
- 资源 A/B 页面和对应的受保护 API。
- 特权测试账号的幂等初始化。
- 注册与登录限流、结构化日志和健康检查。
- 单元测试、API 集成测试、浏览器 E2E 和生产冒烟测试。
- Docker 本地运行与 CloudBase Run 镜像部署。
- 固定评测集、真实模型评测命令及 README 调优记录。

### 2.2 明确不做

- 邮箱或短信验证。
- 找回密码和修改密码。
- 社交登录。
- 后台管理和在线角色编辑。
- 多租户、资源上传、复杂审计后台。
- 微服务拆分。

## 3. 技术方案

### 3.1 应用架构

采用 TypeScript 模块化单体，使用 npm workspaces 管理以下边界：

- `apps/web`：React + Vite 页面。
- `apps/api`：Fastify HTTP 服务。
- `packages/contracts`：Zod 请求、响应和共享类型。

前端构建产物由 Fastify 提供，生产环境只生成一个多阶段 Docker 镜像。模块化单体用于保持认证、审核、资源授权之间的边界，同时避免三天项目引入分布式部署复杂度。

### 3.2 技术栈

- Node.js 22、TypeScript。
- React、Vite、React Router。
- Fastify、Zod。
- Drizzle ORM、`mysql2`、MySQL。
- Argon2id 密码哈希。
- Vitest、Fastify Inject、Playwright。
- Docker、Docker Compose。
- CloudBase Run 上海地域。
- 腾讯云托管 MySQL，优先使用同地域 TDSQL-C MySQL Serverless；若账号区域暂不可用，则切换同地域 TencentDB for MySQL，应用层仍使用相同 MySQL 协议和迁移。
- DeepSeek OpenAI-compatible API，保留通过环境变量切换 Qwen-compatible API 的能力。

### 3.3 生产部署

浏览器通过 CloudBase Run 默认 HTTPS 域名访问单个应用服务。应用通过同地域私有网络连接托管 MySQL，并通过公网 HTTPS 调用模型提供商。应用容器保持无状态，不在容器文件系统保存业务数据。

本地通过 Docker Compose 启动应用和 MySQL。生产环境只部署应用镜像，数据库独立托管，避免 CloudBase 实例重建或扩缩容造成数据丢失。

## 4. 模块职责

### 4.1 Auth 模块

- 注册、登录、退出登录。
- 密码哈希和校验。
- 创建、校验和撤销会话。
- 读取当前用户。

Cookie 只保存高熵随机会话令牌；数据库只保存令牌哈希。Cookie 使用 `HttpOnly`、`Secure` 和 `SameSite=Lax`，会话设置固定有效期。退出登录删除数据库会话并清除 Cookie。

### 4.2 Moderation 模块

- 规范化并校验候选注册名。
- 构造带版本号的社区规则提示词。
- 调用统一的 `ModerationProvider` 接口。
- 校验结构化模型结果。
- 记录审核元数据和失败原因。

DeepSeek 和 Qwen 适配器只能依赖统一接口，业务服务不读取厂商特有响应。

### 4.3 Authorization 模块

权限矩阵集中定义，不在路由和页面中散落角色判断：

| 角色 | 资源 A | 资源 B |
| --- | --- | --- |
| MEMBER | 允许 | 禁止 |
| PRIVILEGED | 允许 | 允许 |

前端根据后端返回展示锁定状态，但前端隐藏或禁用按钮不构成安全边界。资源 API 每次都执行服务端身份和权限校验。

### 4.4 Resources 模块

- 资源 A 返回社区入门指南与公告 Mock 数据。
- 资源 B 返回社区内部运营指标 Mock 数据。
- 不承担认证实现，只声明所需权限。

## 5. 数据模型

### 5.1 `users`

- `id`：UUID 主键。
- `username`：用户展示名。
- `normalized_username`：NFKC 规范化后的唯一用户名。
- `password_hash`：Argon2id 哈希。
- `role`：`MEMBER` 或 `PRIVILEGED`。
- `created_at`、`updated_at`。

### 5.2 `sessions`

- `id`：UUID 主键。
- `token_hash`：会话令牌哈希，唯一索引。
- `user_id`：用户外键。
- `expires_at`、`created_at`。

### 5.3 `moderation_attempts`

- `id`：UUID 主键。
- `candidate_hash`：候选名哈希；拒绝的原始候选名不持久化。
- `provider`、`model`、`prompt_version`。
- `decision`：`ALLOW`、`BLOCK` 或 `ERROR`。
- `category`、`reason`。
- `latency_ms`、`error_code`、`created_at`。

被允许的用户名最终保存在 `users` 表；审核表用于可观测性，不保存模型 API Key 或完整模型原始响应。

## 6. API 与错误语义

| 方法与路径 | 行为 | 主要状态码 |
| --- | --- | --- |
| `POST /api/auth/register` | 审核用户名并创建 MEMBER | 201、400、409、422、429、503 |
| `POST /api/auth/login` | 创建会话 | 200、400、401、429 |
| `POST /api/auth/logout` | 撤销会话 | 204 |
| `GET /api/auth/me` | 返回当前用户 | 200、401 |
| `GET /api/resources/a` | 返回资源 A | 200、401 |
| `GET /api/resources/b` | 返回资源 B | 200、401、403 |
| `GET /health/live` | 进程存活检查 | 200 |
| `GET /health/ready` | 数据库就绪检查 | 200、503 |

服务端错误采用稳定的机器码和简短中文消息。登录失败不区分“用户不存在”和“密码错误”。模型原始响应、数据库错误和堆栈不得返回浏览器。

## 7. 注册与 LLM 审核流程

1. 校验用户名长度和允许字符，执行 Unicode NFKC 规范化。
2. 检查规范化用户名是否已存在。
3. 将候选名作为 JSON 数据交给带版本号的系统提示词，明确禁止把候选名内容当指令执行。
4. 以 `temperature=0` 请求模型返回 JSON：`allowed`、`category`、`reason`。
5. 使用 Zod 严格解析返回值。
6. 允许时，在一个数据库事务中创建 MEMBER 用户和审核记录。
7. 违规时返回 422，并提供简短、可操作但不泄露内部提示词的原因。
8. 网络错误、超时、模型 5xx 或非法 JSON 只重试一次；仍失败则记录 `ERROR` 并返回 503，不把系统故障伪装成“社区违规”。

社区违规类别固定为：辱骂、仇恨歧视、色情、暴力恐怖、违法内容、冒充官方、提示词注入和其他明确破坏社区安全的内容。普通昵称、轻微情绪表达和不确定内容默认允许，以减少误杀。长度和字符等确定性约束由本地校验负责，不依赖 LLM。

注册和登录按 IP 与用户名维度限流，控制撞库风险和模型调用费用。

## 8. LLM 调试与优化方案

仓库提供固定评测集和显式评测命令 `npm run evaluate:moderation`。评测集至少包含 40 个带期望标签的样例，覆盖：

- 中文和英文正常昵称。
- 辱骂、仇恨、色情、暴力、违法内容。
- 冒充管理员或官方账号。
- 空格、大小写、全半角和常见字符变体。
- 把提示词注入文本伪装成用户名的输入。
- 容易误杀的边界案例。

评测报告记录模型、提示词版本、时间、允许/拒绝结果、误杀、漏放、耗时和错误。真实模型评测不进入普通 CI，避免费用、限流和非确定性造成流水线随机失败。

调优过程必须实际执行：

1. 运行 Prompt v1 并保存基线结果。
2. 分类查看误杀、漏放和结构化输出失败。
3. 只针对观察到的问题修改规则、少量示例或输出约束，生成 Prompt v2。
4. 对同一评测集复测并比较结果。
5. 在 README 中写出初始方案、代表性样例、实际误判、具体修改、前后数据、剩余局限和心得。

禁止在 README 中虚构准确率、模型表现或调试过程。Unicode 同形字符、上下文缺失和模型漂移作为已知局限明确说明。

## 9. 页面与交互

### 9.1 登录与注册

使用同一张卡片切换登录/注册。注册时明确提示“注册名将由 AI 根据社区规则审核”，调用期间展示进度。违规、重复、限流和模型不可用使用不同提示。

### 9.2 资源控制台

登录后同时展示资源 A 和资源 B：

- MEMBER 的资源 A 显示“已授权”，资源 B 显示“无权限”并保持可见。
- PRIVILEGED 的 A/B 均显示“已授权”。
- MEMBER 实际请求资源 B 时必须收到后端 403，页面显示权限提示。

页面支持窄屏布局、键盘焦点、显式表单标签和可读错误提示，不投入复杂动画和主题系统。

## 10. 初始化与秘密管理

特权账号由幂等命令初始化。用户名和密码来自生产环境变量，密码哈希后写入数据库。测试账号密码通过交付信息单独提供，不写入公共仓库。

`.env.example` 只包含变量名和无敏感示例。DeepSeek/Qwen API Key、数据库连接串、Cookie 密钥和生产账号密码不得提交 Git 或写入浏览器日志。

## 11. 测试策略

### 11.1 单元测试

- 用户名规范化和本地校验。
- 模型 JSON 解析、重试和错误分类。
- 使用 FakeProvider 验证允许、拒绝和不可用分支。
- 密码哈希和会话令牌处理。
- MEMBER/PRIVILEGED 权限矩阵。

### 11.2 API 集成测试

- 合法注册创建 MEMBER。
- 违规用户名不创建用户并返回 422。
- 模型不可用不创建用户并返回 503。
- 规范化用户名重复返回 409。
- 未登录访问资源返回 401。
- MEMBER 访问 A=200、B=403。
- PRIVILEGED 访问 A=200、B=200。
- 退出登录后旧会话失效。

集成测试使用隔离的 MySQL 测试库和 Fastify Inject，不调用真实 LLM。

### 11.3 浏览器与生产验证

- Playwright 覆盖注册、登录、A/B 权限和退出登录主路径。
- 构建并运行生产 Docker 镜像进行容器冒烟测试。
- 部署后检查健康接口，并分别使用普通账号和特权账号验证授权矩阵。
- 提交前通过非开发设备或移动网络验证公网 URL。

开发期间维护事实日志，记录本人耗时、AI coding 工具、token 估算、Prompt 变更和评测结果；README 只汇总实际记录。

## 14. 风险与降级

- CloudBase/VPC/托管 MySQL 若在 Day 1 阻塞超过两小时，停止新增功能并优先恢复公网健康检查；仅在已有可用 HTTPS 域名或入口时，保留同一应用镜像并切换腾讯云轻量服务器、Docker Compose 与 Caddy。HTTP 明文登录不能作为最终交付。
- 模型提供商异常时注册返回 503，已存在用户的登录和资源访问不受影响。
- Qwen 作为配置级替代提供商，不在业务代码中增加分支。
- 若时间不足，优先保留安全授权、真实模型审核、自动化测试和公网部署，删除视觉润色，不删除 README 调优证据。

## 15. 验收标准

- 公网 HTTPS URL 在中国大陆网络可访问。
- 新注册用户能登录、访问 A，并被后端禁止访问 B。
- 提供的特权账号能访问 B。
- 明显违规注册名被真实 LLM 审核拒绝；模型故障返回可区分的 503。
- 资源 B 不能通过直接调用 API 绕过授权。
- Docker 镜像可构建，本地启动文档可复现。
- 自动化测试通过，生产主路径完成冒烟验证。
- README 完整覆盖题目五项说明和真实的 LLM 调试/优化心得。
